1. SYSTEM OVERVIEW
================================================================================

LightJSC is an intelligent face detection and matching system that ingests
AI face metadata from i-PRO cameras via RTSP ONVIF MediaInput protocol.
The system matches detected faces against watchlists (local or Active Guard)
and dispatches face events to webhook subscribers with real-time notifications.

Key Features:
  • Real-time RTSP metadata ingestion from IP cameras
  • Face detection and template extraction
  • Cosine similarity-based face matching
  • Dual watchlist support (local PostgreSQL or SQL Server Active Guard)
  • Webhook event dispatch with HMAC authentication
  • Web dashboard for monitoring and management
  • REST API for system configuration and queries
  • Background worker pipeline for async processing
  • Prometheus metrics for observability
  • Map visualization with geolocation tracking

4. DATA FLOW / PROCESSING PIPELINE
================================================================================

4.1 FACE DETECTION FLOW
───────────────────────

Source: RTSP Camera (i-PRO with AI metadata)
                ↓
RTSP Metadata Receiver (CameraConnector)
  • Establishes TCP connection to camera RTSP port
  • Streams ONVIF MediaInput metadata
  • Decodes face detection frames
                ↓
Metadata Parser (OnvifParser)
  • Extracts face bounding boxes
  • Extracts face templates (embeddings)
  • Extracts confidence scores
  • Associates with camera and timestamp
                ↓
Ingest Pipeline - FaceEvents Channel
  • Enqueues parsed FaceEvent objects
  • Atomic counter tracking
  • Non-blocking (drops if capacity exceeded)
                ↓
Face Detection Storage
  • Persists to FaceEvents table in PostgreSQL
  • Creates searchable index
  • Stores face image best-shot (if enabled)
                ↓
Dashboard/API
  • Face events queryable via REST API
  • Visible in web dashboard

4.2 FACE MATCHING FLOW
──────────────────────

FaceEvent from Pipeline
                ↓
MatchingWorkerService
  • Dequeues FaceEvent from pipeline
  • Loads active watchlist (cached in memory)
  • Performs cosine similarity matching
  • Applies configurable confidence threshold
  • Per-person threshold support
                ↓
Similarity Calculation
  • Query: face template from detected face
  • Targets: watchlist face templates
  • Algorithm: Cosine similarity (dot product)
  • Result: Vector<VectorMatch> with scores
                ↓
Decision Generation
  • Creates FaceMatchDecision object
  • Enqueues to Decisions channel
  • Records match confidence scores
  • Marks matched person (if any)
                ↓
Webhook Dispatch
  • WebhookDispatcherService dequeues decision
  • Generates JSON event payload
  • Computes HMAC-SHA256 signature
  • POSTs to each subscribed webhook endpoint
  • Retry logic: exponential backoff
  • Failed events → DLQ (Dead Letter Queue)

4.3 WATCHLIST SYNCHRONIZATION
──────────────────────────────

WatchlistSyncService (Scheduled Background Task)

Source Selection: appsettings.json "Watchlist:Source"

If Source = "Local":
  • Queries local Persons + FaceTemplates from PostgreSQL
  • Loaded into in-memory watchlist
  • Synced every N seconds (e.g., 20 seconds)
  • Full refresh every M minutes (e.g., 10 minutes)

If Source = "ActiveGuard":
  • Queries SQL Server Active Guard database
  • Reads from configured read-only connection
  • Synchronizes face data periodically
  • Falls back to local cache on failure

Watchlist Cache:
  • In-memory dictionary for fast lookup
  • Thread-safe access during matching
  • Atomic swap on full refresh

5.1 POSTGRESQL (Local Watchlist & Events)
──────────────────────────────────────────

Primary Data Store | Default: "ipro_face_ingestor"

Tables and Purposes:
  • cameras
    - RTSP connection credentials
    - Authentication for i-PRO cameras
    - Status and last connection time
    - Fields: cameraId, ipAddress, rtspUsername, rtspPassword, rtspProfile, etc.

  • persons
    - Watchlist entries
    - Employee/subject profiles
    - Fields: personId, code, firstName, lastName, category, isActive, etc.

  • face_templates
    - Face embeddings/vectors
    - One-to-many with persons
    - Fields: templateId, personId, embedding[], confidence, source, etc.

  • face_event_records
    - Detected face events
    - One per face detection from cameras
    - Fields: eventId, cameraId, timestamp, templateVector[], confidence, etc.

  • subscribers
    - Webhook subscriptions
    - Filtering rules per subscriber
    - Fields: subscriberId, webhookUrl, secret, enabled, categories, etc.

  • dlq_messages
    - Dead letter queue for failed webhooks
    - Retry tracking
    - Fields: messageId, subscriberId, payload, retryCount, lastError, etc.

  • runtime_states
    - System state persistence
    - Last sync timestamps
    - Watchlist version tracking

  • map_layouts
    - Map visualization configuration
    - Camera positions on map

  • map_camera_positions
    - Geolocation of cameras
    - Latitude/longitude/floor information

Indexes:
  • Primary keys on all tables
  • Foreign key constraints
  • Composite indexes on frequently queried fields
  • Full-text search on person names
  • Vector similarity indexes (if PostgreSQL pgvector extension used)

5.2 SQL SERVER (Active Guard - Read-Only)
──────────────────────────────────────────

Optional Integration | Requires: Active Guard deployment

Purpose: Read-only watchlist synchronization

Typical Tables Accessed:
  • Personnel/People table
  • Face template/enrollment table
  • Department/category information

Connection: Configured in "ConnectionStrings:ActiveGuard"
Authentication: SQL Server credentials
Access Level: Read-only (enforced via database roles)

Sync Mechanism:
  • Dapper for lightweight querying
  • Periodic full synchronization to local PostgreSQL
  • Caching to reduce database load

6. EXTERNAL INTEGRATIONS
================================================================================

6.1 i-PRO CAMERAS
──────────────────

Protocol: RTSP + ONVIF MediaInput
Connection: TCP to camera RTSP port
Metadata: AI face detection frames in metadata stream

Flow:
  1. Camera configured in system with RTSP credentials
  2. CameraConnector established persistent TCP connection
  3. Receives ONVIF metadata frames
  4. Parses face detection and template data
  5. Enqueues to ingest pipeline

6.2 WEBHOOK SUBSCRIBERS
────────────────────────

Protocol: HTTP REST (POST)
Authentication: HMAC-SHA256

Event Payload (face event):
  {
    "faceEventId": "uuid",
    "cameraId": "CAM01",
    "timestamp": "2026-03-18T10:30:45Z",
    "matchedPersonId": "person-uuid" or null,
    "matchConfidence": 0.95,
    "faceImageUrl": "https://...",
    "location": { "latitude": 10.5, "longitude": 106.5 }
  }

Security:
  • HMAC-SHA256 signature in X-Webhook-Signature header
  • Shared secret per subscriber
  • Timestamp validation (prevent replay attacks)

Reliability:
  • Retry up to N attempts
  • Exponential backoff (2^retry seconds)
  • Failed deliveries stored in DLQ
  • Manual retry via admin API

────────────────────────────────────

  • No built-in API authentication (assumed behind reverse proxy/load balancer)
  • WAF/authentication delegated to infrastructure layer
  • CORS policy configured for map endpoints

  • RTSP camera credentials encrypted with AES-GCM

─────────────────

  • Encrypted at rest (AES-GCM)
  • Never logged or exposed in API responses
  • Only decrypted for camera connection

Webhook Secrets:
  • HMAC-SHA256 signing key per subscriber
  • Used for event authentication

Database:
  • PostgreSQL connection string in user secrets
  • SQL Server credentials in user secrets
  • No credentials in source control

8.3 HTTPS / TLS

11. KEY DESIGN PATTERNS
================================================================================

11.1 ASYNC PIPELINE (Channel-based)
────────────────────────────────────

  • Non-blocking work queues
  • Two-stage processing: Events → Decisions
  • Drop overflow (no backpressure)
  • Lock-free with atomic counters
  • Multiple producers, multiple consumers

11.2 REPOSITORY PATTERN
────────────────────────

  • Abstraction over data access
  • Entity Framework Core for PostgreSQL
  • Dapper for SQL Server (lightweight)
  • Unit of work implied (single DbContext)

11.3 OPTIONS PATTERN
──────────────────────

  • Strongly-typed configuration sections
  • Dependency injection friendly
  • IOptions<T> for singleton config
  • IOptionsMonitor<T> for hot reload

11.4 BACKGROUND WORKER ABSTRACTION
─────────────────────────────────────

  • IHostedService implementation
  • Graceful shutdown handling
  • Dependency injection for services

11.5 VECTOR/SIMILARITY INDEXING
──────────────────────────────────

  • Abstraction for swappable implementations
  • Brute-force: Full comparison (high latency, high accuracy)
  • Approximate: Indexed search (low latency, approximate accuracy)
  • Cosine similarity metric

16. INCIDENT MANAGEMENT
================================================================================

Failure Scenarios:

1. Camera Disconnection:
   - CameraConnector retries with exponential backoff
   - Status tracked in RuntimeState
   - Alert via PipelineWatchdog

2. Watchlist Sync Failure:
   - Continues with last known watchlist (cached)
   - Retry on next interval
   - Alert in logs/monitoring

3. Webhook Delivery Failure:
   - Moves to DLQ (Dead Letter Queue)
   - Manual retry available
   - Admin notified via monitoring

4. Database Connectivity Loss:
   - Health check marks service as not ready
   - Requests return 503 Service Unavailable
   - Automatic reconnection attempts

5. Pipeline Stall:
   - PipelineWatchdog detects lack of metadata
   - Alert triggered
   - Potential camera issues investigated

Recovery:
  • Graceful degradation where possible
  • Circuit breaker patterns (external dependencies)
  • Manual intervention via admin API
  • DLQ replay for webhook failures
